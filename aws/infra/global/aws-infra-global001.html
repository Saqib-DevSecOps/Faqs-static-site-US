1. An Overview of the AWS Global Infrastructure Components [AWS: Infrastructure: Components, Regions, Availability Zones, Local Zones, Global Network]
    1. The AWS Global Infrastructure includes the components listed below, roughly in order of scope; from a Region, which is a geographic area with a full complement of AWS services, available in a highly-available manner across multiple Availability Zones, to Points of Presence, which allows content providers to cache their content closer to the end-user:
      1. Regions
        1. The AWS Cloud infrastructure is built around AWS Regions and Availability Zones. An AWS Region is a physical location in the world which has multiple Availability Zones.
        2. An AWS Region is a geographical area, that has multiple AWS physical data centers, which are grouped into multiple logical data centers (called Availability Zones), that serve a full-complement of core and other AWS services, made available across multiple Availability Zones.
          3. Note: The above description distinguished Regions from Local Zones, which are extensions of Regions, and host a more focused set of AWS services.
        1. An AWS Region is "AWS’ term for a physical location where it clusters data centers – that are deployed within a highly available, fault tolerant, and scalable environment."
        2. Each AWS Region exists wholly within a single country's boundary
        3. The benefit of an AWS Region includes, it allows customers to deploy resources closer to end-users, and to meet any data governance laws, which require data to be located in a specific geographic area, such as a single country boundary.
        4. Amazon Regions are designed to be completely isolated from each other. This achieves the greatest possible fault tolerance and stability.
      1. Availability Zones
        1. An Availability Zone (AZ) is a subset of an AWS Region
        2. An AWS Region consists of multiple Availability Zones.
        3. An Availability Zone can be considered a logical data center as each AZ consists of one or more discrete data centers, each with redundant power, networking, and connectivity, housed in separate facilities.
        4. The AZs are physically separated from each other to create a fault domain, in the event of a disaster, yet highly connected to each other within the Region, to enable customers to deploy highly available architectures such as primary, and standby databases across multiple AZs, leveraging the high-speed connection between AZs to perform synchronous replication between the databases
        5. A benefit of AZs is that it enables multi-AZ deployments, where a customer can deploy a service across multiple Availability Zones to minimize or eliminate a single point of failure
        6. Availability Zones enable customers to create an production architecture that is highly available, fault tolerant, and scalable in a way that would not be possible with a single data center.
        7. Within a Region, Availability Zones are connected through low-latency links.
        8. AWS provides customers with the flexibility to place instances and store data within multiple geographic regions as well as across multiple Availability Zones within each AWS Region.
        9. The resilience features of AZs include
          1. Each Availability Zone is designed as an independent failure zone.
          2. Availability Zones are physically separated within a typical metropolitan region.
          3. Each is located in lower risk flood plains (specific flood zone categorization varies by AWS Region).
          4. Each has discrete uninterruptable power supply (UPS) and onsite backup generation facilities.
          5. Each is fed via different grids from independent utilities to further reduce single points of failure.
          6. Availability Zones are all redundantly connected to multiple tier-1 transit providers.
        1. Each Availability Zone is designed as an independent failure zone.
        1. References:

          2. Amazon Web Services (AWS) Data Center Locations: Regions and Availability Zones (dgtlinfra.com)
      2. Local Zones
        1. "Local Zones are an extension of an AWS Region that provide you the ability to place resources in multiple locations closer to your end users."
        2. It is a "a new type of AWS infrastructure designed to run workloads that require single-digit millisecond latency, like video rendering and graphics intensive, virtual desktop applications."
          3. Note: Local Zones are different from Availability Zones in that Local Zones provide a core set of services to end-users who are in geographic proximity and have latency sensitive workloads, while Availability Zones provide access to the full array of AWS services.
        1. "A Local Zone is represented by a Region code followed by an identifier that indicates the location, for example, us-west-2-lax-1a."
        2. The benefit of AWS Local Zones is that it provides a subset of the full-array of AWS services to users in geographic proximity to the Local Zone, with extremely low-latency needs.
        3. Local Zones are designed to serve specific geographies and industries. They contain services targeted to specific industries, e.g. the Local Zone in Los Angeles provides services targeted at the entertainment industry

      1. Points of Presence
        1. In AWS, the Points of Presence consists of Edge Locations and Regional Edge Caches
        2. Edge Locations cache content closer to the end-user.
        3. A content provider, such as a website administrator, or a streaming media service, may choose to place a copy of their content at Edge Locations, giving users near the Edge Location, low-latency access to the content.
        4. Content providers use the AWS CloudFront service to populate Edge Locations
        5. Regional Edge Caches are not used directly by content providers. They are considered mid-tier cache, sitting between your Origin server and the globally distributed Edge Locations. They effectively act as backups to Edge Locations.
        6. Regional Edge Caches are fewer in number than Edge Locations, however each holds more content (has a larger cache-width) than the Edge Locations. The design is such that, if content is aged out of the Edge Location, chances are good that the content may still be available in the Regional Edge Cache, where it can be served to the customer
        7. Origin server is the term for the original location of the content. An origin server stores the original, definitive version of your Object, before it is copied to the Edge Location. Your files, also known as objects, typically include web pages, images, and media files. It can be anything that can be served over HTTP.
        8. If you are serving content over HTTP, your origin server is either an Amazon S3 bucket or an HTTP server, such as a web server. CloudFront gets the object from there and distributes them from the CloudFront edge locations
        9. If previously cached content is no longer available in the Edge Location or the Regional Edge Cache, it will be copied (again) from the Origin server to repopulate the Edge Location, and Regional Edge Caches.
        10. You specify Origin servers, like an Amazon S3 bucket or your own HTTP server, from which CloudFront gets your files which will then be distributed from CloudFront edge locations all over the world.
        11. Your HTTP server can run on an Amazon Elastic Compute Cloud (Amazon EC2) instance or on a server that you manage; these servers are also known as custom origins.
        12. "To deliver content to end users with lower latency, Amazon CloudFront uses a global network of 310+ Points of Presence (300+ Edge locations and 13 regional mid-tier caches) in 90+ cities across 47 countries."
        13. Edge Locations are physical sites whose main purpose is caching content, closer to the end-user. As such, it provides only a subset of AWS services, just what is needed to securely and efficiently make content available to end-users.
        14. CloudFront distribution, tells CloudFront information about how to serve content, including origin servers, regional coverage, etc.
        1. CloudFront edge locations are grouped into geographic regions, which are in turn grouped into price classes. Customers can choose a price class that includes all global list of edge locations, or a subset of  geographic regions.
        2. A subset of services available in Edge Locations include: Amazon Route 53 (DNS queries), AWS Shield (for DDOS protection), AWS Web Application Firewall (to protect web infrastructure from common web exploits), AWS Certificate Manager (to provide custom certificates), and Lambda@Edge (to run your code closer to your end-users)

        1. AWS network backbone - fully redundant, 100GbE, global private fiber

        3. "A fully redundant 100GbE fiber network backbone, often providing many terabits of capacity between Regions."
